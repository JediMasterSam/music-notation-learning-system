# Architecture Sprint 0.1 Review Report

Status: Complete — proposed for Product Owner review  
Architecture baseline reviewed: 0.2  
Review scope: experimental notation and learning workbench amendment

## 1. Executive summary

Architecture Sprint 0.1 amends the Sprint 0 baseline so Prototype 1 is an experimental workbench rather than a demonstration of one rendering treatment.

The amended design preserves canonical musical authority and introduces three separate versioned artifact families:

1. reusable `LearningTransformationDefinition` objects that produce derived `LearningPlan` objects whose chunks reference canonical content;
2. declarative `RepresentationRecipe` objects that compose independent representational strategies without source-code edits;
3. `ExperimentDefinition` and immutable run-manifest objects that reproduce comparisons while keeping human observations separate from automated output.

The revised Sprint 1 is implementation ready. It requires two functional treatments from the same canonical melody fixture—explicit beat grid and proportional spatial melody—and one reusable learning transformation applied to two compatible arrangements. No final notation, pitch mapping, duration encoding, chunking strategy, or learner-facing default has been selected.

## 2. Product-direction conformance

### Canonical authority

Conforms. `Arrangement` owns musical realization only. It contains no learning chunks, recipes, experiment definitions, layout, or workbench state. All transformations are immutable and hash-verified.

### Learning transformation ownership

Conforms. Learning chunks belong to a derived plan generated by a pinned deterministic transformation. Plans use canonical IDs/spans/selectors and reject copied event payloads.

### Representational workbench

Conforms. Recipes independently select time, pitch, duration, label, overlay, disclosure, accessibility, and renderer strategies where compatibility permits. Existing primitives can be recombined through JSON without TypeScript changes.

### Spatial-melody experiment

Conforms. Sprint 1 includes a real proportional treatment with onset x, duration extent, and pitch y invariants, compared to an explicit-grid baseline using the same canonical fixture.

### Reproducibility

Conforms. Recipes, strategies, transformations, experiments, canonical inputs, resolved options, tool versions, diagnostics, and output hashes are pinned in manifests. Reruns must be byte-identical.

### Interaction requirement

Resolved. Sprint 1 uses declarative files, CLI discovery/validation/run commands, and generated static comparison pages. This allows configuration without source edits. A local browser control panel/live preview remains an explicit adapter boundary and is deferred. The previous static-only assumption is narrowed to the Sprint 1 proof, not retained as a full Prototype 1 constraint.

### Compatibility and truthful failure

Conforms. Capability evidence is derived from validated music. Status is `supported`, `supported-with-limitations`, `incompatible`, or `unavailable`. No strategy or transformation may silently approximate, invent data, or fall back.

## 8. Risks and mitigations

### RISK-01 — Configuration matrix explosion

Mitigation: finite strategy kinds, pinned descriptors/options, capability evidence, compatibility statuses, no runtime arbitrary plugins, and pairwise/property test matrices.

### RISK-02 — Learning plan becomes shadow arrangement

Mitigation: reference-only selectors, schema rejection of musical payloads, canonical hash checks, deterministic regeneration, and plan verification.

### RISK-03 — Spatial treatment hardens into product vocabulary

Mitigation: experimental/comparison status in descriptors and output, no default recipe, output-language tests, and independent replaceable strategies.

### RISK-04 — Static interaction proves architecture but frustrates use

Mitigation: Sprint 1 gathers practical feedback on recipe-file/CLI workflow; `workbench-web` boundary is explicit. Browser controls can be added without changing artifact schemas or semantics.

### RISK-05 — Capability vocabulary becomes accidental learner terminology

Mitigation: capability IDs remain internal technical identifiers. Any learner-facing terminology requires Product Owner decision/corpus justification.

### RISK-06 — Deferred renderer breadth masks semantic regressions

Mitigation: H-C/C-D contract fixtures remain required; full-pipeline specificity and canonical distinction tests cannot be deferred.

### RISK-07 — Reproducibility depends on environmental text layout

Mitigation: Sprint 1 semantic coordinates do not depend on font metrics; layout environment/rounding is pinned; accessible text is separate from geometric authority.

### RISK-08 — Experiment artifacts imply learning proof

Mitigation: definitions record research intent/tasks; run manifests record software output; human observations are separate and no effectiveness claim is generated.
